## Proposed change
File as P0 CRITICAL. Reason: the source repo sw.js has a JavaScript syntax error that prevents the entire Service Worker from registering. The production site is currently safe because it uses a DIFFERENT sw.js (generated by the strangler build), but:
1. If anyone runs `npm run build` and deploys from source, the SW breaks completely
2. The `sw-dist-readiness-audit.js` script also fails to parse PRECACHE_ASSETS correctly (gets corrupted data)
3. Any CI/CD pipeline that validates sw.js from source will either fail or produce wrong results

## Evidence
```
$ node -e "new Function(require('fs').readFileSync('sw.js','utf8'))"
SyntaxError: Unexpected string

# Two malformed entries in PRECACHE_ASSETS:
# [3] /css/nagornaya-mobile-toc.css,  ← comma inside string
# [18] /js/series-cards.js,           ← comma inside string

# Production is different and correct:
$ diff <(curl -s https://gospod-bog.ru/sw.js) sw.js | head -5
< ...correctly formatted, 29 assets...
> ...malformed, 24 valid + 2 corrupted...
```

## Why this matters
The sw.js file is the foundation of the site's offline support, precaching, and cache-first strategies. A syntax error in this file means:
- No offline support
- No precaching of critical CSS/JS/fonts
- No stale-while-revalidate for HTML pages
- No cache-first for static assets
- All of the above are currently working in production ONLY because the production sw.js is different from source

This is a ticking time bomb — the next deploy from source will break everything.

## Suggested fix
1. Copy the production sw.js back to the source repo (quickest)
2. OR find and fix the minifier/build step that generated the broken version
3. Add a CI check: `node -e "new Function(require('fs').readFileSync('sw.js','utf8'))"` must pass
