## Deep Dive 1: Service Worker Architecture — CACHE_VERSION vs PRECACHE

### The Problem

`sw.js` is a single minified JS file. It is managed by:
1. `cache-bust.js` — updates CACHE_VERSION string and inline asset hashes
2. `scripts/sw-dist-readiness-audit.js` — reads baseline and compares
3. Manual minification — file is compiled, not source

Three separate mechanisms touch the same file, none knows about the others' changes.

### CACHE_VERSION Lifecycle

```
2026-07-02: baseline created → v182 (gb-v182-gill-toc-actions-20260702)
2026-07-02: sw.js updated → v183
2026-07-03: sw.js updated → v184 (pagefind bootstrap, d5c65647)
2026-07-03: sw.js updated → v185 (runtime no-undef fixes)
2026-07-03: sw.js updated → v186 (gill rail frame, e2f0ae4e)
2026-07-03: sw.js updated → v187 (baptisty root parity, 932af3f3)
```

v183→v187 happened via `cache-bust.js` (5 versions), but baseline was never re-synced. The audit tool warns but doesn't fail.

### PRECACHE Drift

PRECACHE_ASSETS hardcodes 26 assets as an inline JS array. After Pass 51-56 (search lazy loading), `/data/search-manifest.json` and `/js/search.js` became interactive lazy-load — but the PRECACHE wasn't updated.

```javascript
// sw.js (minified) — PRECACHE_ASSETS entry 21 and 22
"/data/search-manifest.json",
"/js/search.js",
```

Both are already in the precache. SW install time is longer than necessary. Users who never use search still get search.js on install.

### Why no automated guard?

`audit-pro.js` checks CACHE_VERSION format but not baseline drift. `sw-dist-readiness-audit.js` has `--require-cache-bump` but it's not a CI gate — only used when explicitly called.

### Fix path

Option A (quick): After every `cache-bust.js` run that updates CACHE_VERSION, automatically re-sync the baseline file.

Option B (correct): Generate sw.js from a JSON manifest (`sw-manifest.json`) at build time. Single source of truth. `sw-dist-readiness-audit.js` reads the manifest. No hand-edited minified JS.

## Deep Dive 2: CSS Architecture — Why 490 !important in floating-cluster.css

### Source

`css/floating-cluster.css` (2881 lines, ~106KB minified) is the primary CSS for PremiumControls (Gill series, Baptist GBS2, floating-cluster UI). It contains:

- Floating tooltip system
- PremiumControls cluster (speed pill, chapter nav, TOC)
- PlayEmber controls (audio player overlay)
- Gillian series submenu
- Responsive mobile overrides

### Why so many !important?

The file implements complex overlay positioning with:
- Mobile touch interactions
- Viewport clamping for tooltips
- Nested media queries (27× `@media (hover: hover)`, 6× `@media (max-width: 899px)`)
- Layer system with `data-fc-*` and `data-gbs2-*` selectors

Each rule competes with site.css and component-level styles, requiring `!important` to win specificity battles. The result: 490 `!important` declarations.

### What audit-pro.js checks

`audit-pro.js` checks only `css/site.css` (18 !important). The ceiling is 202. site.css is a "primary" CSS layer and managed carefully.

floating-cluster.css is "secondary" CSS — not checked. But it has **24×** more !important than site.css.

### Why no CSS refactor?

The CSS code review Pass 68-70 covered floating-cluster.css (confirmed in matrix), but the specific count of 490 !important was either not measured or measured but not entered into the matrix as a distinct P1 item.

## Deep Dive 4: validate.js Warnings — SEO Metadata Drift

### 2 Articles with title ≠ og:title

```javascript
// validate.js — warning is not a hard error
⚠️  [20-antisovetov-pastoru] <title> ≠ og:title
   <title>: "20 антисоветов пастору: как разрушить служение"
   og:title: "20 антисоветов, как пастору разрушить своё служение"

⚠️  [rimlyanam-7-veruyushchiy-ili-neveruyushchiy] <title> ≠ og:title
   <title>: "Римлянам 7: верующий или неверующий?"
   og:title>: "Римлянам 7: верующий, неверующий или человек под законом?"
```

### Analysis

Both are grammar variations, not semantic errors:
- "антисоветов пастору" vs "антисоветов, как пастору" — comma changes meaning slightly
- "верующий или неверующий" vs "верующий, неверующий или человек под законом" — og:title is more precise

### Why not errors?

validate.js treats this as warning (not error) because:
1. SEO impact is ambiguous — `<title>` and `og:title` serve different purposes
2. Both are valid Russian
3. The check is there to catch genuinely different content (e.g., wrong article title vs og:title)

### Recommendation

Add a severity tier: if the difference is a simple punctuation change, warn; if the difference is a content change (different article), error.

For these 2 articles: recommend fixing to make both identical, since the og:title is more complete.

## Deep Dive 5: Commit History Patterns — What They Reveal

### Pattern 1: Micro-commits

2026-07-04 had many `chore: auto-update meta, cache-bust [skip ci]` commits — 1-2 per hour. These are auto-generated by indexnow.yml after each content push. Pro: ensures cache-bust always fresh. Con: makes `git log --oneline` noisy, makes bisect harder, fills history with identical messages.

Count on 2026-07-04: ~8 such commits in the first half of the day alone.

### Pattern 2: Lane naming

All substantive commits follow `[LANE lane/...]` format. Good discipline. Lane policy is respected.

### Pattern 3: Merge commits

`merge:` commits (e.g., `8c318010 merge: seo-fix-og-images lane`) show lane-based workflow. Each merge corresponds to one lane. All merges verified green on CI before merge.

### Pattern 4: Double-fix regression

629ed89a (remove orphaned image files) → fc5f94bd (repair broken references from incomplete orphan cleanup). The fix created a regression, immediately fixed. This is a **code quality signal**: the image cleanup script had incomplete reference checking.

### Pattern 5: Metadata-only vs genuine fix (NEW-59)

c0ab48fc was merged as "fix(seo): correct hard-texts og image dimensions" but was actually metadata-only. Reopened → 6cc68586 genuine fix. This pattern — one agent's "fix" being incomplete — is the same reason AuditRepo exists: multi-witness verification prevents metadata-only fake fixes.
