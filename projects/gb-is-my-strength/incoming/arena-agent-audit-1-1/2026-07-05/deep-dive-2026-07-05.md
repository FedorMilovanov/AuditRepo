## Deep Dive 1: IndexNow Architecture — What the YAML Actually Does

### The Full Picture

```
indexnow.yml (metadata/gate workflow):
  → update-meta.js
  → cache-bust.js
  → validate:static-publication:light (34 checks)
  → git commit cache-bust
  → git push
  → if fails: workflow conclusion = failure

deploy.yml (deploy workflow, triggered by workflow_run):
  → runs on success OR failure of indexnow.yml
  → Checkout
  → npm ci
  → cache-bust.js (NOT SKIPPED on workflow_run trigger!)
  → full validation (37 checks)
  → Build + Pagefind
  → Visual parity
  → Deploy to GitHub Pages
  → Submit to IndexNow (Bing + Yandex) ← step 208
```

### Key Revelations

1. **IndexNow submission IS in deploy.yml** (step "Submit to IndexNow (Bing / Yandex)", lines 208-235). This is the correct design. Search engines are notified AFTER successful deployment.

2. **Cache-bust IS NOT SKIPPED** when triggered via workflow_run:
   ```yaml
   if: github.event_name != 'workflow_run'  # ← condition is FALSE on workflow_run
   run: node scripts/cache-bust.js           # ← cache-bust RUNS
   ```
   The comment says "skip if IndexNow already did it" but the condition logic is inverted from the comment. On workflow_run trigger, cache-bust runs regardless of whether IndexNow succeeded or failed.

3. **Deploy condition includes 'failure'** — deploy proceeds when IndexNow metadata/gate fails. This is the intended design per the comment: "Deploying anyway because deploy.yml runs its own cache-bust and validation."

### The Real Risk

If indexnow.yml fails due to a CONTENT issue (e.g., MDX article has broken frontmatter), deploy.yml will:
1. Still deploy the content (with its own cache-bust ✅)
2. Still run the full 37-check gate ✅
3. Still submit to IndexNow ✅

**But:** The metadata update (lastmod timestamps, generatedAt) from indexnow.yml is NOT applied. The deploy will use the LAST successful indexnow.yml's metadata. This means sitemap.xml lastmod dates may be stale for the next deploy cycle.

**Severity re-assessment:** AUDIT-P2-DEPLOY-ALWAYS is NOT P2 — it's P3 INFO. The design is intentional, the safeguards work, and the risk is acceptable.

### Root Cause Analysis

`floating-cluster.css` implements the PremiumControls overlay system for:
- Gill v16 desktop/mobile rail controls
- Baptist GBS2 controls
- Floating tooltip/reflow system
- PlayEmber audio player

Each feature competes with site.css and inline styles. `!important` is used to win specificity wars across:
- Mobile vs desktop
- Touch vs pointer
- Overlay open vs closed states
- Theme (light/dark) variants

**Not a bug** — this is the cost of a complex overlay system in a non-component-framework environment. The real question is whether the file should be modularized (split by feature) so each module can have its own ceiling.

## Deep Dive 4: OG Image vs LCP Priority — SEO vs Performance Trade-off

### The Four Mismatched Routes

| Route | og:image | LCP (fetchpriority=high) | Semantic |
|-------|----------|--------------------------|----------|
| `/articles/20-antisovetov-pastoru/` | og-20-antisovetov-pastoru.webp (1200×630) | mirror.webp (article hero) | Different images |
| `/articles/kod-da-vinchi/` | og-kod-da-vinchi.webp | hero-kod-da-vinchi.jpg | Different images |
| `/` | og-preview-1200x630.webp | (none — no fetchpriority=high) | N/A |
| `/pastor-series/` | og-hero.webp | (none) | N/A |

### Analysis

For social sharing (Facebook, Telegram, WhatsApp):
- og:image determines the preview card
- LCP image is irrelevant

For Core Web Vitals LCP metric:
- fetchpriority=high image determines LCP score
- og:image is irrelevant

**These are separate optimization goals.** The mismatch is intentional and correct for this site's audience.

**When they SHOULD align:** When the hero image of an article is both the most visually prominent content AND the social share preview. For `/`, using og-preview for og:image is correct (generic brand card). The LCP image being og-nagornaya-propoved is fine — it loads fast but isn't the social card.

**For kod-da-vinchi:** The article hero (hero-kod-da-vinchi.jpg) might actually be a BETTER og:image than the generic og-kod-da-vinchi.webp. Consider updating the og:image to match the LCP image for better social engagement.

**Recommendation:** For article routes (kod-da-vinchi, 20-antisovetov), consider making og:image = hero image. For index/landing routes, current setup is fine.

**Key insight:** SEO-05 (og:image dimensions) uses `warn()` not `err()`. So any route with non-1200×630 og:image passes with a warning. This is why hard-texts OG image was "fixed" (c0ab48fc) as metadata-only — seo-audit.js would only warn, not fail.

## Deep Dive 6: check-workflows.js vs actionlint — Complementary Tools

### check-workflows.js scope (202 lines)

Checks:
- Workflow `name:` present
- Workflow `on:` present  
- Workflow `permissions:` present
- `validate:static-publication` includes `page-ownership:check`
- `strangler:*` scripts include `page-ownership:dist`
- deploy.yml uses Node 22+, `npm ci`, `validate:static-publication`
- deploy.yml has `concurrency:` and `actions/upload-pages-artifact@v3`
- dist deploy includes `pagefind:build:dist`, `visual:parity:production`

**What it does NOT check:** YAML expression syntax, shell command syntax, secret variable names.

### actionlint scope (would catch)

- Expression syntax errors: `${{ secrets.FOO }}` where `secrets.FOO` undefined
- Missing `inputs.` in composite actions
- Shell syntax errors in `run:` commands
- Invalid `uses:` paths

**Conclusion:** check-workflows.js is a POLICY contract checker. actionlint is a SYNTAX correctness checker. Both needed. The finding AUDIT-P2-ACTIONLINT-NOT-WIRED remains valid — actionlint should be added to CI, but it complements check-workflows.js rather than replacing it.
