Executive summary
Репозиторий уже живёт в дисциплине Astro/strangler с жёсткими правилами для агентов: сначала
обязательное чтение WORK_MODES.md, AGENTS.md, LANE_LOCK_POLICY, затем сверка с 
route-migration-matrix.json; для production-impact работ обязательны финальные прогоны
npm run validate:static-publication и npm run guard:shared-files. В 
package.json закреплён Node >=22.12.0, а основной статический барьер уже включает
большой набор parity-, content-, readability-, MDX- и ownership-аудитов. Это значит, что rollout
PremiumControls нельзя делать как “ещё один виджет”, его нужно встраивать как маршрутно-
типизированную систему, которая уважает migration-mode каждого route и уже существующую CI-
архитектуру. 
Ключевая инженерная ошибка прошлых попыток предсказуема: смешение двух разных задач — 
что рисуем и где это стоит. Новый PremiumControls не должен сам решать геометрию через 
position: fixed; геометрию должен решать отдельный anchor-слой. MDN прямо различает 
absolute и fixed: absolute позиционируется от ближайшего positioned ancestor , а fixed
— от viewport. Поэтому для standalone-статей задача не “приклеить новый cluster справа”, а 
заменить старый control новым внутри старого layout slot, сохранив визуальный якорь у
хлебных крошек. 
Архитектурно rollout должен завершиться тремя стабильными примитивами: 
PremiumControlAnchor отвечает за размещение, PremiumControls — за SVG/UI-вариант, 
premium-controls-controller.js — за поведение. CSS должен стать единым canonical source;
Astro допускает и scoped styles, и <style is:global>, но ?raw/ ?url описаны как advanced/
not recommended, а статические <link> в Astro корректно работают только с абсолютными
путями из /public или внешними URL. Поэтому правильная цель — не держать две ручные
копии CSS, а иметь один source и один детерминированный способ доставки в native Astro и при
необходимости в публичный CSS-asset. 
По маршрутам проект уже явно делится на несколько archetype-групп. Матрица маршрутов
показывает, что /articles/hermenevticheskaya-otsenka-hristotsentrichnoy-
germenevtiki/ уже находится в strict-native с target mode mdx-native-article, а Gill-
статьи пока вообще исключены из общего non-excluded migration scope через articles/dzhon-
gill-*. Одновременно legacy-shadow-app маршруты вроде /karty/*, /map/, /konfessii/
russkij-baptizm/ требуют другого обращения и не должны получить article controls по
умолчанию. Это и задаёт rollout-логику: сначала два пилота, потом — раскатка по archetype-
типам, а не по произвольным страницам вручную. 
Минимальный итоговый критерий очень простой. После rollout на standalone-статьях day/night
control должен стоять у breadcrumbs-level, а не у viewport-right; PlayEmber должен открывать
speed-panel и сохранять скорость; не должно быть duplicate-controls; все [data-fc-action]
должны кликаться; CI должен оставаться зелёным на текущем пакете gates; readTime 35 vs 50

Цели и жёсткие ограничения
Цель rollout — полностью заменить разрозненные legacy theme/play/save/search controls единой
системой PremiumControls, но не ломая migration contract проекта. Это значит: для каждого route-
type нужно заранее определить, есть ли у него вообще право на article controls, какой вариант
controls нужен и какой anchor управляет положением. У standalone strict-native статей цель —
заменить старый article control slot премиальным SVG-кластером без потери геометрии. У Gill-
подобных series pages цель — специализированный rail/mobile-bar вариант. У app/routes и
landing/catalog routes цель обратная — ничего не внедрять, кроме явно разрешённых future
exceptions. Такой подход прямо следует из того, что route matrix уже отличает strict-native, 
mdx-native-article, native-main-with-legacy-chrome и legacy-shadow-app, а AGENTS/
WORK_MODES требуют учитывать route mode до начала работы. 
Есть четыре жёстких constraint-а, которые нельзя нарушать. Во-первых, rollout должен работать в
существующей CI-дисциплине: Node >=22.12.0, astro:build, 
strangler:build:production-like, validate:static-publication, page-ownership, 
pagefind, visual:parity:production, sw:dist:audit:deploy-switch. Во-вторых, CSS не
может оставаться в состоянии “компонентная копия + отдельная ручная копия”, иначе drift
неизбежен. В-третьих, нельзя оставлять смешанную семантику у PlayEmber: если control
интерактивен, он не должен одновременно изображать disabled purely via aria-disabled; MDN
подчёркивает, что aria-disabled="true" только сообщает состояние, а подавление
функциональности разработчик обязан обеспечивать сам. В-четвёртых, rollout нельзя строить на
голом fixed там, где важен старый anchor . 
Acceptance criteria должны быть формализованы до кодинга, иначе слабый агент опять уйдёт в
“CI green, но визуально криво”. Для standalone article acceptance — это сохранение визуального
слота старого control, а не просто наличие нового DOM-узла. Для Gill acceptance — rail и mobile-
bar должны быть частью route chrome, а не отдельным плавающим FAB. Для site-wide acceptance
— контрольный список должен измерять поведение, а не только наличие классов: alignment,
clickability, persistence, отсутствие дублей и зелёная сборка. Так как Astro scoped styles всегда
приходят поздно в каскаде относительно импортов той же specificity, а is:global отключает
scoping, rollout обязан осознанно управлять CSS order и тем, какой способ доставки стилей
выбран для PremiumControls.

Целевая архитектура внедрения
Правильная система должна делиться на три слоя.
PremiumControlAnchor — чисто layout-компонент. Его единственная работа — занять старый
visual slot и отдать внутри него место для controls. У standalone articles этот anchor должен жить в
том же DOM-slot, где раньше стоял #themeToggle или тематически эквивалентный legacy
control. У Gill этот anchor отдельный: rail-foot на desktop и mobile-bottom-bar на mobile. Anchor не
знает про Play panel, bookmark logic или hotkeys. Он знает только geometry, z-index, safe-area и
responsive breakpoints.
PremiumControls — presentation/UI-layer . Он рендерит theme/search/font/play/save SVG controls,
но не решает абсолютное местоположение страницы. Внутри него допустимы variant-ы single, 
series-lite, gill-rail, gill-mobile. Именно здесь должны жить визуальные токены
вроде --ember-size, gap, radius, control-size, glow, pulse, dark-mode variants.
premium-controls-controller.js — runtime-layer . Он инициализирует все roots, реализует
click delegation, speed panel, persistence, keyboard shortcuts, close-on-outside/Escape,
синхронизацию active state и optional legacy-bridge hooks.
Такой split обязателен не только ради чистоты. Он напрямую решает ключевую ошибку
прошлого: если control layer сам включает position: fixed, он немедленно отрывается от
старой выверенной геометрии. MDN прямо говорит, что fixed ориентируется на viewport, а 
absolute — на ближайший positioned ancestor . Следовательно, anchor должен наследовать
старую absolute geometry, а control surface внутри anchor должен быть position: static или 
relative.

CSS-архитектура и runtime contract
Главное правило rollout: один canonical CSS source. Не два ручных файла, не “временный Astro
style + копия в /css”, а один источник. Astro подчёркивает, что scoped styles и <style 
is:global> — разные механики, а ?raw/ ?url CSS imports — advanced и “not recommended for
most users”; для статических <link> требуются absolute paths из /public или внешние URL.
Значит, правильная архитектура — хранить PremiumControls CSS в одном source внутри Astro
source tree и либо импортировать его как основной source для native routes, либо build-step-ом
эмитить derived public CSS для тех редких legacy/static contexts, где нужен именно 
<link href="/css/...">. Ручное дублирование нужно прекратить. 
Рекомендуемая структура такая:
src/styles/premium-controls.css           # canonical source
src/components/premium-controls/*.astro   # components import canonical CSS
public/css/premium-controls.css           # generated artifact only if really

Если legacy/static HTML действительно нуждается в публичном CSS-файле, он не должен
редактироваться руками. Он должен генерироваться на build/deploy этапе из canonical source. Тем
самым исчезает класс проблем “в компоненте одно имя класса, в public CSS — другое”.
Отдельный обязательный слой — проверка @layer. MDN прямо сообщает, что normal
declarations вне слоя всегда имеют больший приоритет, чем normal declarations внутри named
layers. Если PremiumControls CSS живёт внутри @layer components, а старые unlayered selectors
в site.css или mobile-hotfix.css продолжают касаться .gb-*, .gb-save, .gb-
ember, .gb-theme-toggle или legacy equivalents, то даже при корректном import order controls
могут внезапно проиграть. Поэтому rollout должен сделать две вещи одновременно: либо
держать PremiumControls вне layers, если это системный layer-top CSS; либо ввести явный audit на
unlayered overlap по ключевым selectors. 
CSS contract для PremiumControls должен быть токенизирован. Минимальный набор:
:root{

--pc-gap: 8px;
--pc-radius: 18px;
--pc-z-anchor: 32;
--pc-z-panel: 36;

--pc-safe-right: env(safe-area-inset-right, 0px);

Главная идея: размеры не захардкожены внутри SVG-компонентов. --ember-size и shape
tokens управляются variant-ом. Это снимает класс проблем “в Gill ember не виден / слишком
большой / разъехались controls”.
На runtime-layer обязательны пять контрактов.
Первый — инициализация всех roots, а не первого. Если на странице есть desktop root и mobile
root, controller обязан делать document.querySelectorAll('[data-fc-
root]').forEach(initRoot). Это уже не optional refinement, а базовое требование
архитектуры, потому что Gill archetype по определению multi-root. 
Второй — data-fc-mode обязателен. Нельзя по умолчанию считать root single.
Минимальный enum: single, series-lite, gill, disabled. Это избавляет от ложных body
classes и route-specific side effects.

Третий — aria-disabled contract. MDN подчёркивает, что aria-disabled="true" не
отключает поведение само по себе; разработчик обязан вручную его подавить. Поэтому нельзя
одновременно держать PlayEmber семантически disabled и открывать по клику speed panel. Для
rollout есть только два честных варианта: либо PlayEmber реально disabled и не кликается, либо
он интерактивен и тогда должен иметь интерактивную семантику, например aria-
haspopup="menu"/ dialog, aria-expanded, aria-controls. Для текущего
пользовательского требования рациональнее второй вариант: на маршрутах без аудио
PlayEmber работает как control “настройки скорости/будущей озвучки”, а не как disabled button. 
Четвёртый — pointer-events. Если внешний decorative shell использует pointer-events: none
для пропуска кликов сквозь фон, то интерактивные потомки и speed panel обязаны получать 
pointer-events: auto. MDN отдельно предупреждает, что pointer-events: none
исключает сам элемент как target, а дочерний subtree targetable только при явном override. Это
критично для speed panel и маленьких SVG-кнопок. 
Пятый — localStorage. Для site-wide rollout нужен один canonical key-space и backward-compatible
aliasing. Рекомендуемый contract:
gb:theme              # текущая тема или явный alias к существующему site key
gb:bookmark:v1        # сохранённые статьи / alias bridge к текущей bookmark 
системе
gb:audio:rate         # скорость прослушки
gb:controls:variant   # optional debug/feature-flag state
Если в репозитории уже есть другие ключи, controller должен читать и старые, и новые, но
писать уже в canonical namespace. Это избавляет от “pilot drift”, когда один route хранит скорость
под одним ключом, а другой — под другим.
Event flow для PlayEmber должен быть формализован:
click ember
→ stop propagation from generic action if panel-open flow handles it
→ open panel adjacent to same root
→ set aria-expanded=true
→ highlight current rate from localStorage
→ on rate click: persist value, add pulse/glow, optionally emit custom event
→ wait 280–380ms for feedback animation
→ close panel and restore aria-expanded=false
→ also close on Escape, outside click, route change, root unmount
Если route действительно поддерживает аудио позже, этот же flow расширяется start/pause
actions; если route аудио не поддерживает, speed preference всё равно честно живёт как
preference UI, а не как faux-disabled action.

Тестирование, acceptance и CI
Проверки должны быть минимальны, но хирургически точны. Уже существующий gate велик, и
это хорошо: его не нужно ломать, его нужно дополнить. package.json уже фиксирует основной
стек: validate:static-publication, astro:build, strangler:build:production-like, 
page-ownership, visual:parity:production, pagefind, sw:dist:audit. Значит,
внедрение PremiumControls должно входить в существующую дисциплину как дополнительный
contract layer , а не как отдельный “ручной прогон если вспомним”. 
Первый барьер — HTML. Сначала нужно убрать битые хвосты/атрибуты, затем подключить 
html-validate. Его CLI принимает files/directories/globs, умеет formatter-ы json, stylish, 
checkstyle, а с --max-warnings 0 даёт non-zero exit на предупреждениях. Это идеальный
следующий слой после простых grep/regex-детекторов мусора. 
Второй барьер — accessibility smoke. После того как HTML стал валиден, axe-core можно
встроить в Playwright/browser smoke. Сам axe-core позиционируется как engine для automated
Web UI testing и интеграции в существующие functional/integration/browser tests. Следовательно,
его стоит включать не до HTML validation, а после неё, на двух-трёх критических маршрутах
пилота.

Rollback strategy должен быть build-safe и route-local. Лучший вариант — route allowlist flag. Пока
archetype не зелёный, route может держать premiumControls: false или equivalent build-time
feature flag, а legacy controls остаются каноническими. Для каждого pilot/phase сохраняется
возможность быстро вернуться к старому DOM output без касания shared CSS/JS на всем сайте.
Rollback не должен означать “откатить весь canonical controller”; он должен означать “выключить
new controls на конкретном archetype route”.
На уровне audit migration rollback особенно важен. Если validator раньше ожидал old marker , а
route уже переведён, audit должен временно принимать оба варианта, пока rollout phase не
закрыта. Только после зелёного archetype-phase allowed set сужается до нового exact marker
contract. Это убирает соблазн слабому агенту временно “починить CI” возвратом к старому
marker-name вместо реального исправления rollout.
Рабочая инструкция для слабого агента
Слабому агенту нельзя стартовать с хаотичного “поправлю CSS”. Он должен сделать очень
конкретную последовательность.
Сначала среда. node --version должен быть не ниже 22.12.0, потому что это
зафиксировано в package.json. Затем npm ci. Для локальной сборки нужен npm run 
astro:build; для production-like dist — npm run strangler:build:production-like.
Финальный release barrier — npm run validate:static-publication, а после production-like
build — npm run page-ownership:dist:production-like, npm run pagefind:build:dist, 
npm run visual:parity:production, при необходимости npm run sw:dist:audit:deploy-
switch. Всё это уже формализовано в package.json и AGENTS discipline.

full gate.
Итоговый contract внедрения
Если свести весь план к одной фразе, она такая: PremiumControls должны стать route-aware
control system, а не ещё одной плавающей штукой. На article-breadcrumb routes система
обязана уважать старый visual anchor и заменить старый control slot новым SVG block без потери
геометрии. На gill-rail routes она обязана быть частью route chrome. На app/routes и
landings она обязана уметь не появляться вовсе. Именно такой rollout соответствует и migration
matrix проекта, и Astro CSS-модели, и существующей CI discipline. 
Если агент не может объяснить, какой archetype он сейчас меняет, какой anchor он сохранил, 
какой exact mode стоит у root, и как новый CSS избегает drift между Astro и public asset
delivery, он ещё не готов делать rollout. Если он говорит только “CI green”, но не может показать
alignment у breadcrumbs-level, отсутствие duplicate-controls, сохранение speed selection и route-
correct variant choice, значит задача не закрыта.
