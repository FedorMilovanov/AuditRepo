## 2. Алгоритмическое ограничение древа генеалогии (Single-Parent Restriction)

Интерактивное древо генеалогии Священного Писания (`/rodosloviye/`, реализованное на ReactFlow в `src/components/genealogy/GenealogyTree.tsx`) — один из самых красивых разделов сайта. Однако хирургический анализ алгоритма раскладки вскрыл **серьезное несоответствие между полнотой данных и их визуальным представлением**.

### 2.1 Суть дефекта в `src/components/genealogy/layout.ts`
В базе данных `data/genealogy/genealogy.json` для многих персонажей прописаны оба родителя (и `father`, и `mother`). Например:
*   Исаак: `father: "isaac"`, `mother: "sarah"`
*   Иаков: `father: "isaac"`, `mother: "rebekah"`
*   Соломон: `father: "david"`, `mother: "bathsheba"`
*   Иисус: `mother: "mary"`, `father` (генеалогические линии)

Однако алгоритм раскладки в файле `layout.ts` использует функцию `resolveParent`, которая жестко срезает многоуровневый граф до дерева с **одним родителем**:

```typescript
function resolveParent(p: Person, ids: Set<string>): string | null {
  if (p.father && ids.has(p.father)) return p.father;
  if (p.mother && ids.has(p.mother)) return p.mother;
  return null;
}
```

И в логике построения ребер (`edges`):
```typescript
  const edges: Edge[] = [];
  for (const p of filtered) {
    const parentId = resolveParent(p, ids);
    if (!parentId) continue;
    ...
```

### 2.2 Последствия для визуальной точности
1.  **Потеря материнских линий:** Из-за того, что `resolveParent` возвращает `father` в приоритетном порядке, ребро от матери к ребенку **никогда не строится**, если у ребенка указан отец. Линии Сарры, Ревекки, Вирсавии и других великих библейских праматерей визуально «висят в воздухе» — от них нет стрелок к детям, хотя в данных эта связь прописана. 
2.  **Смерть двойного родословия (Матфея vs Луки):** Писание дает две разных генеалогических ветви Христа (по линии Иосифа в Мф и Марии в Лк). В данных это заложено, но визуализатор не способен отобразить обе линии одновременно к одному узлу Иисуса из-за ограничения `resolveParent -> 1 parent id`.
3.  **Причина костыля:** Библиотека `dagre`, используемая для авто-раскладки, плохо справляется с циклами и множественным родительством (multi-parent DAGs), часто ломаясь или накладывая узлы друг на друга. Предыдущие агенты решили проблему радикальным упрощением графа до древовидной структуры, лишив пользователя библейской полноты данных.

## 3. Проблема "Двух Миров" Гилла (The `gbs-` vs `gbs2-` Mismatch)

При переносе серии "Джон Гилл" на Astro проект раскололся на два несовместимых стандарта верстки, сосуществующих параллельно на страницах одной и той же серии.

### 3.1 Сопоставление структуры классов
Давайте сопоставим классы нативного Astro-шаблона Гилла (`src/components/article-pilots/gill-context/GillContextPageChrome.astro`) и легаси-шаблонов Частей 1/2/3 (`src/components/article-pilots/gill-part1/GillPart1PageChrome.astro`):

| Элемент интерфейса | Нативный мир (v16 Context) | Легаси-мир (Части 1/2/3) |
| :--- | :--- | :--- |
| **Атрибут области видимости** | `data-gill-v16="context"` | *Отсутствует* |
| **Боковая панель (Desktop)** | `class="gbs-rail"` | `class="gbs2-rail"` |
| **Футер боковой панели** | `class="gbs-rail-foot"` | `class="gbs2-rfoot"` / `class="gbs2-rhead"` |
| **Кнопки управления** | `gbs-rail-foot__btn` (32px) | `gbs2-mctl` (38px, огромные) |
| **Мобильная нижняя панель** | `class="mobile-bottom-bar"` | `class="gbs2-bbar"` |
| **Панель оглавления (Mobile)** | `class="toc-overlay"` (Bottom Sheet) | `class="gbs2-sheet"` (Tabbed Dialog) |
| **Изображения в оглавлении** | **Отсутствуют (чистый текст + Roman)** | `gbs2-thumb` (baptisty-style thumbnails) |

## 4. Скрытая нестыковка: Полнота карты сайта (Sitemap Coverage Gap)

Наш глубокий диагностический скрипт (Check 24) выявил серьезную нестыковку в индексации карт:
```text
⚠️  [WARNING] Production-dist sitemap coverage -> Missing sitemap URLs for:
  - /karty/pavel/
  - /karty/shoftim/
  - /karty/melachim/
  - /karty/shvatim/
  - /karty/yeshua/
  - /karty/maccabim/
  - /karty/early-church/
  - /karty/revelation/
```

### 4.1 Суть нестыковки
Эти 8 карт задекларированы в `page-ownership.json` и `route-migration-matrix.json` со статусом `"production-dist"`. То есть они компилируются и публикуются в результирующем дистрибутиве `dist/`.
*   **Реальность:** Однако из-за того, что они фактически являются пустыми страницами-холдингами, они **полностью исключены из поисковой карты сайта `sitemap.xml`**!
*   **Почему это рассинхрон:** Мы наблюдаем доктринальное расхождение на уровне метаданных: `page-ownership.json` утверждает, что это полноценные боевые страницы продакшна, в то время как `sitemap.xml` относится к ним как к закрытым техническим черновикам. Это вводит в заблуждение автоматические скрипты проверки плотности контента.

## 5. Изолированность 3D Баптизма (WebGL Island)

Страница `konfessii/russkij-baptizm/` содержит невероятный интерактивный 3D-движок на Three.js (карта истории баптизма в России).

*   **Архитектурная изоляция:** Приложение абсолютно слепо к контенту сайта. Оно не связано с серией статей `baptisty-rossii` (10 статей). Идеальный "сценарий мечты": при клике на точку временной шкалы в WebGL-приложении должна открываться соответствующая статья, а при чтении статьи — плавающий кластер должен подсвечивать прогресс на 3D-карте. На текущий момент этот мост отсутствует. WebGL — это просто изолированная "игрушка" на полях, не интегрированная в общий контентный граф `links-graph.json`.

## 6. Хирургические рекомендации для будущего рефакторинга

Чтобы окончательно закрыть эти сложные технические долги, необходимо:

1.  **Исправление `resolveParent` в ReactFlow:**
    Переписать раскладку `layout.ts` так, чтобы граф поддерживал множественные ребра (от мамы и папы одновременно), используя кастомную логику группировки родительских узлов на одном уровне Y. Это вернет библейских праматерей на карту генеалогии.
2.  **Завершение миграции Гилла (Unification):**
    Перевести все 4 оставшиеся части Гилла на макет `GillContextPageChrome.astro`. Это автоматически сотрет легаси-семейство классов `gbs2-*`, удалит избыточные картинки-миниатюры из оглавления Гилла и унифицирует мобильный опыт.
3.  **Синхронизация sitemap по статусам:**
    Либо перевести неактивные роуты карт в статус `build-only` в `page-ownership.json` (чтобы они не компилировались в деплой продакшна), либо добавить их в `sitemap.xml`, если владелец решит открыть эти черновики для индексации поисковиками.
